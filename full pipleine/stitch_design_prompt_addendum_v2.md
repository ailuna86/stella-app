# Stitch prompt addendum v2 — Essay Revision (2 screens) + Writing Coach

Paste this after screen 12 in your existing `stitch_design_prompt.md` prompt, or send it as a follow-up message in the same Stitch project so it inherits the design system already established. Reuse the exact same colors, card style, button shapes, and typography defined at the top of that original brief — don't restate the system, just apply it.

Screens 13 and 14 below are drawn directly from two real engine-generated reference documents (`revision_workspace.html` and `revision_ai_comparison_v1_7.html`) — every panel, button, and content block listed actually exists in the product today; nothing here is invented. Those reference files are currently rendered in plain gray/Arial developer styling with no relation to the app's design system — the point of this prompt is to redesign that exact content using ST.ELLA's real visual language.

13. **Essay revision workspace (Gold only).** Reached from a "Revise this essay" button on the feedback report screen (screen 9). Header states the screen's purpose plainly: hints only, no model answer shown yet at this stage.

    A sticky toolbar directly under the header holds four filter chips — **Show all / Only red / Yellow + red / Only yellow** — that filter the paragraph list below to just the sentences matching that status, plus a **"Copy clean essay to editor"** button on the right.

    Below the toolbar, a **"Before you write or revise"** panel (collapsible) shows: a word-count target line ("Aim for at least 250 words. Good practice target: 260-290 words"), an expandable **"Universal IELTS Task 2 structure"** checklist (Introduction: task frame, position, preview of main ideas; Body 1 / Body 2: one main idea, explanation, specific example, link back; Conclusion: summary + final answer, no new example), a **"Body paragraph formula"** as a small numbered list (topic sentence → argument/reason → development → specific example → link back), and two short callouts: "Strong examples: use a specific place/actor/programme/situation, show what happened, link it to the argument" and "Avoid: private family anecdotes or very general examples as the main evidence."

    Next, a **"Revision plan"** panel lists 3-4 short waves in priority order (e.g. "Wave 1 — Check sentences before keeping," "Wave 2 — Rewrite red areas," "Wave 3 — Improve yellow areas," "Wave 4 — Final language pass"), each as its own line with a small colored dot (rose for red-priority waves, amber for yellow-priority waves) instead of using literal emoji, plus one closing note: "Start with red sentences, then improve yellow sentences. Keep green sentences unless you have a clear reason to change them."

    The main body is a **two-column split** (stacks vertically on mobile): left column is the **annotated original essay**, right column is **your revision**.

    Left column — one card per paragraph, each with a colored left border (rose = rewrite, amber = improve, mint = keep), a heading showing the paragraph's role (Introduction / Body / Conclusion) and its overall status label, an optional small warning line if the paragraph's structural role needs checking (e.g. "Check the paragraph role before keeping this paragraph"), then every sentence in that paragraph as its own smaller card tinted the same three-color system with a status badge (Rewrite / Improve / Keep). Each sentence card has a collapsed "Hint" that expands to show the plain-language issue explanation, and — when the engine flagged a specific phrase — the exact quoted phrase inline (in a small code-style chip) next to its explanation, so the student sees precisely which words triggered the flag, not just a general comment.

    Right column — a single large text area pre-filled with the original essay text (via the "copy clean essay to editor" action), where the student writes their revision directly. Below the text area, two actions side by side: a secondary button **"Compare with AI model"** (free, instant) and a primary button **"Resubmit for re-evaluation"** (uses one evaluation credit, shown as a small counter nearby) — make the cost difference between these two actions visually clear so students don't confuse them.

14. **Essay revision — AI comparison (Gold only, appears after "Compare with AI model" is used).** Either a new state on the same screen or a slide-in panel below the editor — the student shouldn't have to navigate away. A banner at the top states the key rule plainly: "This model rewrite is generated from your **original** essay, not your revision — use it to see both against a strong version. Don't copy it as your own work," plus the model essay's total word count once generated.

    Below the banner, one **comparison card per paragraph**, each showing the paragraph's role as a heading, then a **three-column layout** (stacks to one column on mobile): **Original** | **Your revision** | **AI model rewrite** — the AI column visually distinguished with a light mint tint so it reads as the "target" reference, the other two neutral. Under the three columns, a short **"Why this is stronger"** note (amber-tinted callout) lists 2-3 short bullets explaining what the model paragraph does better, plus — when available — a one-line "Example design" note describing the specific example the model used. A collapsible **"Useful lexical upgrades"** list below that shows `weaker phrase → stronger phrase` pairs pulled from the student's own original essay, each with a short reason.

    At the very bottom, a collapsed **"Show full AI model essay"** expander reveals the complete model essay as continuous readable text, with the same anti-copying reminder repeated once more.

15. **Writing coach mission (Gold only, two states).** Reached from a "Writing coach" teaser card on the Gold dashboard (screen 7) and/or a "Coach" nav item. Keep the tone encouraging and low-stakes — this is a short daily exercise, not a test.

    **State A — today's mission.** A highlighted violet header card (matching the Gold-tier card style used elsewhere) shows the mission title, a one-line description of today's focus skill, a weekly-focus label, a streak goal ("Complete 3 missions this week"), and a time estimate badge ("10 min"). Below it, the mission detail: the specific goal for today in plain language, a short numbered list of steps, a set of "stimulus" prompts shown as small individually-numbered cards the student responds to, and a compact checklist ("Before you submit, check…"). Below that, a response text area for one numbered answer per item, with placeholder text showing the expected format, and a primary "Submit for grading" button. No evaluation-credit counter here — mission grading is fast and unlimited, so the framing should feel closer to a quick quiz than an essay submission.

    **State B — graded result.** Replaces or slides in below the response area once graded. A status badge at top reads Pass (mint), Partial pass (amber), or Not yet (rose), followed by a short plain-language overall comment. Two short lists follow: "What went well" (mint-tinted, checkmarks) and "Fix first" (rose-tinted, bullets) — 2-4 items max each. Below that, each submitted item gets its own small card with the student's sentence, a short issue explanation, and — where available — a suggested revision in a mint-bordered callout. A closing instruction line tells the student what to do next, followed by a secondary "Try again" button that resets the form.

**Design consistency note:** all three screens should reuse the exact same rose/amber/mint three-color status system the student already learned from the feedback report screen (screen 9) — same colors, same meaning (rose = needs rewriting/not yet, amber = needs checking/partial, mint = good/pass) — so nothing here introduces a new visual convention to learn.
